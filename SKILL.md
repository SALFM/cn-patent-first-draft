# 中国专利初稿生成 Skill

本 Skill 用于生成中国发明专利或实用新型专利的初稿文本。输出结果仅作为提交给专利代理师、律师或企业法务继续加工的技术性初稿，不承诺授权成功，不替代专利代理师、律师或法务的专业判断。

生成时必须遵守：不编造现有技术文献，不编造实验数据，不编造附图，不编造申请人、发明人、身份证号、代理机构、优先权、申请号或其他主体/程序信息。缺失信息统一使用 `[待补充]`，并在待确认事项、自检报告或建议补充信息中说明。

## 1. 使用场景

- 用户提供技术交底、研发说明、方案概要、产品结构说明、流程说明、算法说明、附图清单、附图标记或内部申请材料，希望生成中国发明或实用新型专利初稿。
- 用户需要把技术材料整理为权利要求书、说明书、摘要、附图说明、附图标记和自检报告。
- 用户需要初步判断方案更适合发明、实用新型、同日发明加实用新型布局，或存在外观设计/不授权客体边界风险。
- 用户需要对内部材料中的专利名称、发明人、身份证号、代理信息、摘要、应用前景、说明书审阅状态进行一致性检查。
- 用户输入涉及 AI、算法、软件、大数据、推荐、用户画像、图像识别、自动驾驶、医疗预测等，需要把方案组织为技术问题、技术手段和技术效果闭环。

## 2. 不适用场景

- 用户要求直接承诺授权、规避审查、规避法律责任，或要求替代专利代理师、律师、法务出具最终法律意见。
- 用户要求编造现有技术、对比文件、实验条件、实验数据、性能指标、附图、申请人、发明人、身份证号、申请号、优先权、代理机构或代理师信息。
- 方案明显违反法律、社会公德或妨害公共利益，或依赖违法获取/利用的遗传资源。
- 主题本质属于科学发现、智力活动规则和方法、疾病诊断和治疗方法、动物和植物品种、原子核变换方法及相关物质、主要起标识作用的平面印刷品设计，且不能转化为可专利的技术方案。
- 用户需求主要是外观设计图片、照片、简要说明或视觉设计保护，而非发明/实用新型技术方案初稿。

## 3. 输入信息要求

优先要求用户提供以下信息；未提供的字段使用 `[待补充]`：

- 基础信息：拟申请专利名称、拟申请类型、技术领域、具体应用场景、技术交底文本。
- 现有技术：已有方案、最接近现有技术、现有技术缺陷、公开时间和公开方式。
- 技术方案：整体方案、必要技术特征、可选技术特征、结构/步骤/模块/参数/数据流/控制逻辑。
- 技术效果：由技术特征直接产生或合理推出的技术效果；如依赖实验，应提供实验条件、方法、对照对象和数据。
- 实施方式：优选实施例、变形例、替代方案、参数范围、工作原理、工作过程。
- 附图信息：真实附图清单、图号、图名、图示内容、附图标记编号和部件名称。
- 主体和程序信息：申请人、发明人、身份证号或统一社会信用代码、代理机构、代理师、优先权、分案、同日双报、涉密和保密审查状态。
- 内部材料：发明人排序表、知识产权申请承诺书、国防科技大学知识产权申请审批表、专利应用前景证明、专利代理委托书、说明书。
- AI/算法/软件/大数据方案额外信息：具体技术场景、技术问题、数据来源、数据合法性、输入数据、处理流程、模型/算法结构、训练方法、部署环境、模块交互、输出结果、技术效果、敏感数据风险。

## 4. 用户输入不足时的处理规则

- 先区分缺失类型：
  - 阻断型缺失：无法判断专利类型，无法识别技术问题，缺少技术方案，缺少必要技术特征，实用新型无结构对象，AI/算法方案无技术场景或技术效果。
  - 高风险缺失：缺少最接近现有技术、区别特征、实施例支撑、附图、实验数据、主体信息、优先权、数据来源和合法性说明。
  - 可后补缺失：申请人地址、代理机构细节、审阅记录、应用前景等不影响技术文本骨架但影响交付完整性的字段。
- 遇到阻断型缺失时，优先列出具体补充问题；除非用户明确要求先出草稿，否则不强行生成完整专利初稿。
- 用户要求先出草稿时，可生成“基于现有信息的风险稿”，但必须用 `[待补充]` 标记缺失事实，并在待确认事项、自检报告和建议补充信息中列明风险。
- 可以合理概括用户已经提供的事实，但不得新增未给出的部件、步骤、参数、数据、附图、实施例或技术效果。
- 对已公开、优先权、分案、同日双报、涉密、遗传资源、敏感数据等事项，用户未提供时不得默认无风险，应列为待确认事项。

## 5. 专利类型判断规则

1. 先排除禁止性情形和不授权客体：
   - 违反法律、社会公德或妨害公共利益的方案，输出风险提示，不按普通可授权方案生成。
   - 涉及遗传资源时，检查直接来源、原始来源和合法获取/利用情况；缺失时列为待确认事项。
   - 科学发现、智力活动规则和方法、疾病诊断和治疗方法、动物和植物品种、原子核变换方法及相关物质、主要起标识作用的平面印刷品设计，原则上不按发明/实用新型生成；动物和植物品种的生产方法可继续按发明判断。
2. 再识别外观设计边界：
   - 创新点主要是产品整体或局部形状、图案、色彩及其组合，且目的主要是美感或视觉识别时，提示更接近外观设计。
3. 判断是否适合实用新型：
   - 创新点必须落在产品的形状、构造或者二者结合上。
   - 产品应为可由产业方法制造、具有确定形状和构造、占据一定空间的实体。
   - 方法、流程、算法、软件、用途、控制策略、材料配方、分子结构、金相结构、自然形成形状、装饰图案，不作为实用新型核心保护对象。
4. 判断是否适合发明：
   - 产品、方法或者其改进形成的新的技术方案，可按发明组织。
   - 方法流程、制造工艺、控制方法、处理步骤、系统运行方法、材料制备方法、检测方法、设备与方法结合方案，通常按发明处理。
   - 计算机、软件、算法相关方案，只有体现为解决技术问题、采用技术手段并获得技术效果的具体方案时，才按发明组织。
5. 发明和实用新型均可能时：
   - 结构型产品可根据创造性高度、审查周期、保护期限、保护策略判断发明、实用新型或同日双报。
   - 如提示同日发明加实用新型，必须说明同样发明创造只能授予一项专利权；申请时两件申请均需分别说明已申请另一专利；后续发明授权时可能需要放弃已授权且未终止的实用新型。

## 6. T1-T13 槽位映射

| 槽位 | 用户输入字段 | 生成输出字段 |
|---|---|---|
| T1 | 专利名称、核心主题、主题名称 | 专利标题、权利要求主题、说明书首行、摘要名称、全文统一称谓 |
| T2 | 上位技术领域 | 说明书“技术领域”中的上位分类 |
| T3 | 具体技术方向、应用场景或产品类型 | 说明书“技术领域”中的具体领域定位 |
| T4 | 背景技术、已有产品、已有方法、传统方案或公开技术 | 说明书“背景技术”中的现有方案描述 |
| T5 | 现有技术问题、缺陷、限制或不足 | 背景技术末尾的问题引出 |
| T6 | 发明目的或待解决技术问题 | 发明内容/实用新型内容中的目的段 |
| T7 | 整体技术方案、总体流程、系统组成或装置构成 | 发明内容/实用新型内容中的总体方案段 |
| T8 | 核心结构、核心步骤、关键模块、关键算法或关键组合 | 权利要求必要技术特征和说明书核心贡献段 |
| T9 | 有益效果、性能改善或使用效果 | 发明内容/实用新型内容中的技术效果段 |
| T10 | 附图编号、图名、图示内容、附图标记 | 附图说明条目和附图标记表 |
| T11 | 优选实施方式一事实 | 具体实施方式中的第一实施例 |
| T12 | 优选实施方式二、变形例、替代例 | 具体实施方式中的第二实施例或变形例 |
| T13 | 工作原理、操作步骤、运行过程或控制过程 | 工作原理、工作过程、执行流程或控制逻辑 |

生成文本不得保留 `T1-T13`、空括号、二选一模板文本、重复标点或无意义占位符。方法类发明应将“结构”调整为“步骤、流程、算法、控制逻辑或系统模块”。模板中的“批露”必须修正为“披露”。

## 7. 生成流程

1. 解析用户输入，提取技术主题、事实、推断、缺失项、附图、主体信息、程序信息和内部材料字段。
2. 执行专利类型判断，输出发明/实用新型/外观设计边界/不授权客体风险建议。
3. 建立“现有技术缺陷 -> 技术问题 -> 技术方案 -> 技术效果”的闭环。
4. 拆解必要技术特征和可选技术特征，识别哪些特征对技术问题和技术效果有贡献。
5. 生成术语统一表，统一专利名称、技术对象、部件名称、步骤名称、模块名称和附图标记。
6. 规划权利要求结构，先确定独立权利要求主题，再安排从属权利要求层级。
7. 生成权利要求书，确保每项权利要求有说明书支持。
8. 生成说明书，发明使用“发明内容”，实用新型使用“实用新型内容”；有真实附图时生成“附图说明”，无附图时不生成该标题。
9. 生成摘要，压缩为一个连续段落，控制在 300 字以内。
10. 生成附图说明与附图标记；无真实附图时不得虚构图号、图名、图示内容或“如图所示”。
11. 对 AI、算法、软件、大数据类方案执行技术性和数据合规专项检查。
12. 执行内部材料一致性自检和交付前综合自检。
13. 按固定输出格式输出专利初稿生成结果。

## 8. 权利要求生成规则

- 权利要求书以说明书为依据，清楚、简要地限定保护范围。
- 多项权利要求使用阿拉伯数字连续编号，格式为 `1.`、`2.`、`3.`；编号前不写“权利要求”或“权项”。
- 权利要求书内不加标题，不插入附图，通常不使用表格。
- 一份权利要求书至少包括一项独立权利要求，可以包括从属权利要求。
- 独立权利要求应从整体上反映技术方案，写入解决技术问题不可缺少的全部必要技术特征。
- 对技术问题和技术效果有贡献的特征应优先进入独立权利要求或关键从属权利要求；不要堆砌与技术问题无关的常规部件、业务标签或外观描述。
- 从属权利要求应引用在前权利要求，并通过附加技术特征进一步限定；多项从属权利要求只能择一引用在前权利要求，且不得作为另一项多项从属权利要求的引用基础。
- 产品权利要求优先用结构、组成、连接关系、模块交互或可确定参数限定。
- 方法权利要求优先用步骤、流程、操作条件和参数限定。
- 不在同一项权利要求主题中混合产品和方法。
- 功能性或效果性特征应谨慎使用；仅在结构/步骤限定不可行或不如功能/效果限定恰当，且说明书能够直接、肯定验证时使用。
- 附图标记如写入权利要求，应放在对应技术特征后的括号中，并与附图说明一致；附图标记仅便于理解，不限制保护范围。
- 实用新型默认生成“一项独立权利要求 + 若干从属权利要求”，独立权利要求优先采用“两部分式”：`一种……，包括……，其特征是……`。
- 权利要求不得写生产权、销售权、商业目标、广告语、贬低语、原因说明或仅有功能效果的表述。
- 避免“例如”“最好是”“尤其是”“必要时”等导致范围不清的表达；慎用“约”“接近”“高”“低”“强”“弱”等相对词，使用时必须给出明确技术边界。

## 9. 说明书生成规则

- 说明书第一页第一行写明专利名称，且与请求书信息、摘要、权利要求主题名称保持一致；名称前不得冠以“发明名称”“名称”等字样。
- 发明说明书通常包括：技术领域、背景技术、发明内容、附图说明、具体实施方式。
- 实用新型说明书通常包括：技术领域、背景技术、实用新型内容、附图说明、具体实施方式。
- 无真实附图时不生成“附图说明”标题；有真实附图时必须逐图说明。
- 说明书段落编号必须使用 `[0001]`、`[0002]`、`[0003]`……并按实际段落连续编号。
- 说明书正文可以包含化学式、数学式或表格，但不得嵌入附图；流程图、框图、曲线图、相图和照片均作为附图或附图说明处理。
- 说明书必须清楚、完整，以本领域技术人员能够实现为准。
- 每个技术问题应对应至少一个技术方案特征组合和至少一个技术效果。
- 技术方案不得只写功能或效果，必须写明实现该功能或效果的结构、步骤、参数、模块关系、数据流或交互机制。
- 有益效果应由技术特征直接带来或合理推出，采用客观表达，避免商业价值、市场前景或主观体验替代技术效果。
- 具体实施方式应逐项支撑权利要求；产品类实施例写结构、组成、连接关系、配合关系和动作过程；方法类实施例写步骤、条件、参数和顺序。
- 宽范围、上位概括、并列选择、数值范围和功能性限定需要实施例、实验数据或技术理由支持。
- 必须依赖实验结果才能成立的效果，应写明实验条件、测试方法、对照对象和数据；不得虚构实验数据。
- 说明书不得使用商业性宣传用语、无依据贬低他人或他人产品的表达，也不得用“如权利要求……所述”代替具体说明。

## 10. 摘要生成规则

- 摘要写成一个无标题连续段落，不分条。
- 摘要应包含专利名称、所属技术领域、要解决的技术问题、解决问题的技术方案要点和主要用途。
- 摘要包括标点符号在内不得超过 300 字。
- 摘要不得使用商业性宣传语、广告语、卖点介绍、主观夸张或单纯功能性产品介绍。
- 摘要不得加入说明书和权利要求没有支持的新内容。
- 有真实附图时，选择一幅最能说明主要技术特征的说明书附图作为摘要附图，并输出图号；不得虚构摘要附图。
- 实用新型摘要应突出相对于背景技术在形状、构造或其结合上的改进技术特征，并提示应有摘要附图。

## 11. 附图说明和附图标记规则

- 有几幅附图的，按“图 1”“图 2”顺序编号并连续排列。
- 附图说明只基于用户提供或明确存在的真实附图生成；不得编造附图、图号、图名、图示内容、附图标记或“如图所示”表述。
- 附图标记使用阿拉伯数字；说明书文字、附图说明、权利要求中的同一附图标记应指代同一组成部分。
- 说明书文字部分未提及的附图标记不得出现在附图中；附图中未出现的附图标记不得在说明书文字部分提及。
- 附图中除必要词语外不得有其他注释；必要词语应使用中文，必要时可在括号中注明原文。
- 流程图、框图、逻辑框图、工艺流程图应作为附图处理，并在框内给出必要文字和符号。
- 一般不得使用照片作为附图；实用新型附图不得使用照片。
- 实用新型必须有能够表示要求保护产品形状、构造或其结合的附图；无附图属于不予受理风险。
- 初稿应维护附图清单和附图标记表，字段至少包括图号、图名、标记编号、部件名称。

## 12. AI、算法、软件、大数据类发明特别规则

- 先判断是否存在具体技术场景、技术问题、技术数据、技术手段和技术效果。
- 不能仅以“模型”“算法”“平台”“推荐”“画像”“智能”等名词证明技术性。
- 仅涉及抽象算法、数学计算规则、计算机程序本身、游戏规则、纯商业规则或管理规则且不包含技术特征的，提示智力活动规则和方法风险。
- 技术方案应体现：为解决技术问题，采用利用自然规律的技术手段，并获得符合自然规律的技术效果。
- 输入数据应具有具体技术领域中的技术含义；输出应为技术结果，而非单纯商业、金融、营销、管理或主观分类结果。
- 技术效果可体现为精度、效率、可靠性、安全性、存储量、传输量、处理速度、硬件执行效率等客观效果。
- 算法特征、商业规则特征不得被简单割裂；若其与技术特征在功能上相互支持并产生技术效果，应在权利要求和说明书中体现相互作用。
- 说明书应写明模型模块、层级、连接关系、训练步骤、关键参数、输入数据、输出数据、数据预处理、模型与具体应用场景的结合关系、部署环境、系统交互和技术效果验证或合理说明。
- 方法权利要求可按采集、预处理、特征提取、训练、推理、融合、控制、通信、输出等技术步骤撰写。
- 装置、设备、系统、计算机可读存储介质、计算机程序产品权利要求应与方法步骤或模块交互对应，并得到说明书支持。
- 涉及个人信息、隐私数据、生物识别、人脸识别、用户画像、精准营销、推荐决策、自动驾驶伦理、医疗预测、公共场所图像采集、公共安全等，应输出合法来源、授权同意、必要性、处理范围、脱敏、访问控制、存储/传输安全、隐私保护、公共利益、算法公平性和社会公德风险提示。
- 若数据采集、标签管理、规则设置、推荐决策或自动化决策违反法律、社会公德或妨害公共利益，即使包含算法或技术模块，也提示不能授予专利权的实质性风险。

## 13. 内部材料一致性自检规则

仅检查以下六类内部材料，不扩展到未列明材料：

- 发明专利-申请知识产权发明人排序表。
- 发明专利-知识产权申请承诺书。
- 发明专利-国防科技大学知识产权申请审批表。
- 发明专利-专利应用前景证明。
- 发明专利-专利代理委托书。
- 发明专利-说明书。

检查规则：

- 用户未提供标准值的信息，只能写 `[待补充]`，不得自行编造。
- 身份证号必须按文本处理，前导 0 不得丢失；发现位数异常、数字格式、科学计数法或疑似吞 0 时，列为高风险问题。
- 发明人必须为自然人，且应为对发明创造实质性特点作出创造性贡献的人；不得填单位、集体、课题组、实验室、AI、模型或系统名称。
- 发明人必须在发明人排序表、知识产权申请承诺书、国防科技大学知识产权申请审批表之间保持一致，并检查姓名、人数和顺序。
- 专利名称必须在知识产权申请承诺书、国防科技大学知识产权申请审批表、专利应用前景证明、专利代理委托书、说明书之间保持一致；不得使用旧名称、简称或改写名称。
- 摘要应以用户确认的最终摘要为准；多份材料出现摘要时应保持一致，未提供最终摘要时写 `[待补充]`。
- 代理信息应保持一致；代理机构名称应为国家知识产权局登记全称，机构代码准确，并与公章一致；未提供标准代理信息时写 `[待补充]`。
- 专利应用前景证明应检查专利名称和应用前景内容；应用前景缺失、不完整或仍为占位内容时提示补充。
- 说明书应检查名称一致性和是否审阅；没有明确审阅状态或审阅记录时，不得默认已审阅，应写 `[待补充]`。
- 申请人身份信息和联系方式应提示由代理机构或经办人核实。

## 14. 禁止编造规则

- 不得虚构现有技术文献、对比文件、公开日期、公开方式或检索结论。
- 不得虚构实验条件、测试方法、对照对象、实验数据、性能指标、实施例或技术效果。
- 不得虚构附图、图号、图名、图示内容、附图标记或“如图所示”表述。
- 不得虚构申请人、发明人、身份证号、统一社会信用代码、地址、代理机构、代理师、优先权、申请号、分案信息、同日双报状态、审阅结论等主体或程序信息。
- 不得把不能从技术方案合理推出的效果写成确定效果。
- 不得把 AI、单位、课题组、团队、实验室、模型或系统名称写作发明人。
- 不得把商业目标、管理目标、金融预测、营销转化、用户偏好或主观分类直接包装成技术效果。
- 不得保留 `T1-T13`、空括号、二选一模板文本、未填字段或无意义占位符。
- 对确需保留但缺少事实依据的字段，统一使用 `[待补充]`，并说明需要用户确认的内容。

## 15. 固定输出格式

生成专利初稿时，必须严格使用以下固定输出格式和标题顺序，不增删一级或二级标题：

```markdown
# 专利初稿生成结果

## 一、待确认事项

## 二、术语统一表

## 三、权利要求书

## 四、说明书

## 五、摘要

## 六、附图说明与附图标记

## 七、自检报告

## 八、建议用户补充的信息
```

各部分填写要求：

- “待确认事项”：列出专利类型、公开、优先权、分案、同日双报、涉密、遗传资源、主体信息、敏感数据、实验数据、附图等待确认事项。
- “术语统一表”：列出专利名称、技术对象、部件/步骤/模块名称、附图标记、内部材料基准值和统一用语。
- “权利要求书”：仅输出权利要求文本，使用阿拉伯数字连续编号。
- “说明书”：按专利名称、技术领域、背景技术、发明内容/实用新型内容、附图说明、具体实施方式组织；段落编号使用 `[0001]`、`[0002]`、`[0003]`……。
- “摘要”：一个连续段落，300 字以内。
- “附图说明与附图标记”：有真实附图时列图号、图名、图示内容、标记编号、部件名称；无真实附图时写明 `[待补充]` 和缺失风险，不编造。
- “自检报告”：按“风险项 + 影响 + 建议修改/待确认”的方式输出，并覆盖专利类型、名称一致性、主体信息、技术闭环、权利要求支持、说明书充分公开、摘要、附图、AI/算法、内部材料一致性、禁用表达和禁止编造检查。
- “建议用户补充的信息”：给出可执行的问题清单，优先列阻断型和高风险缺失。
