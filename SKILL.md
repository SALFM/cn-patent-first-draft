# 闭环迭代式中国专利草案生成 Skill

## 1. Skill 定位

本 skill 是“闭环迭代式中国专利草案生成 skill”，用于基于用户技术交底、研发说明和可选附图/草图信息，自动生成并迭代优化中国发明或实用新型专利草案。默认工作重点是技术方案撰写、说明书支撑、权利要求布局、摘要和建议附图方案，不生成请求书、发明人表、代理委托书等正式程序文件。

输出结果是提交给专利代理师、律师或企业法务继续加工的技术性草案，不是最终法律意见；不承诺受理、公开、授权、维持有效或规避侵权/无效风险，不替代专利代理师、律师或企业法务的专业判断。

必须遵守：不编造现有技术文献、检索结论、公开日期、实验数据、测试结果、产品结构、参数、已经存在的正式附图或图号标记等事实。申请人、发明人、身份证号、统一社会信用代码、代理机构、代理师、优先权、申请号、分案、同日双报等主体/程序信息默认不进入本技术草案；只有用户明确要求正式申请信息时，才可按用户提供的真实信息照录，仍不得推断或补写。缺失技术事实统一使用 `[待补充]`，或标注为“建议用户确认”。

## 2. 默认闭环工作流与执行形态

本 skill 默认且必须使用结果状态驱动的闭环迭代流程，不提供非迭代的“普通模式”或“单轮初稿模式”。每次生成都应先形成 v0 草案，再进行至少一次 Review、Weakness Routing、Regression Check 和 Iteration Decision，并根据结果状态决定是否继续修复。

默认执行形态是在当前对话中完成闭环迭代。只有当用户明确要求“pipeline mode”“多 Codex”“自动开启新对话”“分阶段执行”或类似表达时，才使用 Pipeline Controller 多 Codex 阶段编排；Pipeline Controller 只是把同一个闭环拆成多个阶段执行，不是替代闭环的普通模式。

除非用户提供的信息完全无法识别任何技术方案，不因信息不完整而停止；应生成“基于现有信息的风险稿”，并列明剩余风险。

1. Phase 0：输入解析与边界判断
   提取技术主题、事实、推断、缺失项、用户保护目标、现有技术、附图/草图信息和公开/合规边界；排查不授权客体、违法/社会公德/公共利益、遗传资源、AI/数据合规等边界风险。
2. Phase 1：发明挖掘与技术闭环构建
   建立“现有技术缺陷 -> 技术问题 -> 技术手段 -> 技术效果”的闭环，识别核心创新点、必要技术特征、可选技术特征和可能的 fallback。
3. Phase 2：技术特征矩阵生成
   生成技术特征矩阵，标明特征类型、必要性、对应问题、对应效果、进入独权/从权/说明书/附图建议的位置、用户依据和风险。
4. Phase 3：权利要求架构设计
   先生成 Claim Scope Ladder，再写正式权利要求；规划独权、从权、产品/方法/系统/设备/介质/程序产品布局，以及实用新型结构化布局。
5. Phase 4：说明书与实施例支撑构建
   构建说明书支撑、实施例、变形例、替代方案、参数范围、模块交互和技术效果说明，为当前权利要求和未来修改空间提供原始记载。
6. Phase 5：附图绘制建议与附图标记建议
   不生成正式附图，不虚构已存在附图；输出建议图号、图名、图示内容、图中应包含的部件/模块/步骤/箭头、建议标记、绘制要点、权利要求/说明书对应关系和用户需确认的技术事实。
7. Phase 6：初稿生成
   生成 v0 权利要求书、说明书、摘要、术语统一表、支持映射和附图绘制建议。
8. Phase 7：多角色专利审查模拟
   使用多角色视角审查当前草案，输出评分、扣分原因、Major/Medium/Minor weaknesses，并判断 weakness 是可由 AI 基于现有材料修复、需要用户补充事实、可接受风险还是阻断型问题。
9. Phase 8：弱点路由
   将每个 weakness 归类到 Weakness Routing Table，生成 routing tickets，明确修改对象、修改指令、验收标准和初始状态。
10. Phase 9：定向修复与重写
   仅修复状态为 Open 且可基于现有材料修复的 ticket；不得修复 Needs User Facts ticket，除非用户已经补充事实。
11. Phase 10：回归检查
   检查修复是否关闭、是否引入新问题、权利要求与说明书/摘要/术语/附图建议是否同步，以及禁止编造规则是否被违反，并更新 ticket 状态。
12. Phase 11：迭代状态决策
   在每轮回归检查后输出 Iteration Decision，判断结果状态和是否继续自我迭代。
13. Phase 12：最终输出
   仅在结果状态为 PASS、NEEDS_USER_FACTS、RISK_ACCEPTED 或 STOP_LIMIT 时，输出新版闭环格式的当前最佳草案、迭代日志、迭代状态决策记录、最终评分卡、剩余风险和待确认事项。

## 3. 迭代控制器 / Iteration Controller

本 skill 不以固定轮数作为闭环目标。迭代次数不是目标，结果状态才是主控制器；最大轮数只是安全上限，用于防止无限循环。

必须按以下规则控制闭环：

1. 先生成 v0 草案。初稿 v0 不计入迭代次数。
2. v0 生成后必须至少进行一次审查判断，但是否进入修复轮次由结果状态决定。
3. 一轮有效迭代定义为：Review -> Weakness Routing -> Targeted Revision -> Regression Check -> Iteration Decision。
4. 每轮结束后必须输出一个结果状态：PASS、REVISE、NEEDS_USER_FACTS、RISK_ACCEPTED 或 STOP_LIMIT。
5. 只有结果状态为 REVISE 时，才进入下一轮定向修复。
6. PASS、NEEDS_USER_FACTS、RISK_ACCEPTED、STOP_LIMIT 均停止自我迭代。
7. 最大安全轮次只限制“修复轮次”，不是执行目标：
   - 快速模式：最多 1 个修复轮次；
   - 标准强度：最多 3 个修复轮次；
   - 深度模式：最多 5 个修复轮次。
8. “最多”不是“必须”。实际修复轮次由结果状态自动决定。
9. 不允许为了凑满轮次而重复审查、重复改写或做无实质提升的文字润色。
10. 不允许为了达到 PASS 状态而编造事实、实验数据、现有技术、公开日期、已经存在的正式附图或任何主体/程序信息；主体/程序信息默认不作为本技术草案的通过门槛。

状态判定优先级：

1. 若继续修复会导致编造事实，或核心问题必须依赖用户真实事实，优先判定为 NEEDS_USER_FACTS。
2. 若存在可基于现有材料修复的 Open Major 或 Medium ticket，且未达到最大安全轮次，判定为 REVISE。
3. 若达到最大安全轮次、连续两轮提升不足或没有实质收益，判定为 STOP_LIMIT。
4. 若无 Open Major，但仍有已披露且不阻断交付的风险，判定为 RISK_ACCEPTED。
5. 只有主要质量门均通过、无编造风险且剩余风险不阻断当前交付时，才判定为 PASS。

每轮结束时，先进行 Regression Check，再进行 Iteration Decision。Iteration Decision 必须输出：

- 当前结果状态；
- 是否继续迭代；
- 继续或停止的原因；
- 若继续，下一轮要处理的 routing tickets；
- 若停止，停止状态和剩余风险。

停止条件不应只看评分。必须同时看：

- Major ticket 是否关闭；
- 是否存在 Needs User Facts；
- 权利要求是否有说明书支持；
- 术语是否同步；
- 附图建议是否覆盖关键特征；
- 是否存在编造风险；
- 是否还有可执行 routing ticket；
- 是否达到最大安全轮次。

## 4. 结果状态定义 / Iteration Result States

### PASS

含义：当前草案已经通过当前信息条件下的主要质量门，可以输出最终草案。

触发条件至少包括：

- 不存在 Open 状态的 Major Weakness；
- 权利要求核心技术特征均有说明书支持；
- 技术问题—技术手段—技术效果闭环成立；
- 独立权利要求没有明显非必要细节；
- 从属权利要求形成合理 fallback ladder；
- 附图绘制建议覆盖关键技术特征；
- 术语在权利要求、说明书、摘要、附图建议中基本一致；
- 未发现疑似编造现有技术、实验数据、产品结构、参数或已存在正式附图等事实；
- 主体/程序信息没有被擅自写入或作为默认阻断项；
- 总评分达到目标分，或在当前用户信息条件下已经达到可交给专利代理师继续加工的状态。

下一步动作：停止自我迭代，输出最终草案、最终评分卡、剩余风险和需用户确认事项。

### REVISE

含义：当前草案未通过质量门，但问题可以基于用户已提供材料继续修复。

触发条件至少包括：

- 存在 Open 状态的 Major 或 Medium Weakness；
- 这些问题有明确 routing ticket；
- 修复可以基于当前用户技术交底、合理上位概括或已有草案完成；
- 修复不需要编造真实事实；
- 下一步修改对象明确，例如权利要求、说明书、摘要、附图建议、术语表、支持映射或格式。

下一步动作：进入下一轮 Weakness Routing -> Targeted Revision -> Regression Check。

### NEEDS_USER_FACTS

含义：剩余问题必须依赖用户补充真实事实，AI 不能继续自主修复。

触发条件至少包括：

- 需要真实实验数据；
- 需要真实测试结果；
- 需要真实产品结构；
- 需要真实现有技术文献、专利号、公开日期或检索结论；
- 需要算法参数、训练数据、模型结构、部署环境或模块交互细节；
- 继续修复会导致编造事实或把未确认内容写成确定事实。

未提供正式附图本身不触发 NEEDS_USER_FACTS；默认输出建议附图方案即可。主体/程序信息缺失也不触发 NEEDS_USER_FACTS，除非用户明确要求生成正式申请文件并要求填入这些字段。

下一步动作：停止自我迭代，输出当前最佳草案，并在“剩余风险与需用户确认事项”中列明需要用户补充的技术事实。

### RISK_ACCEPTED

含义：当前仍存在风险，但风险不是阻断型，且已经明确披露，可以输出当前版本作为代理师继续加工的草案。

触发条件至少包括：

- 剩余问题主要是非阻断风险；
- 风险已经在评分卡和剩余风险中明确披露；
- 当前版本仍然具备完整权利要求书、说明书、摘要和附图建议；
- 继续修改不会明显提升质量，或者需要用户进一步商业判断；
- 适合作为专利代理师、律师或企业法务继续加工的技术草案。

下一步动作：停止自我迭代，输出当前草案和风险接受说明。

### STOP_LIMIT

含义：达到安全上限，或者继续自我迭代没有实质收益。

触发条件至少包括：

- 达到当前模式最大修复轮次；
- 连续两轮总分提升小于 0.3；
- 没有新的可执行 routing ticket；
- 继续修改只是同义改写或格式润色；
- 继续修改可能引入编造风险；
- 继续修改可能破坏已经稳定的权利要求—说明书支持关系。

下一步动作：停止自我迭代，输出当前最佳草案、未关闭问题、停止原因和后续建议。

## 5. Pipeline Controller / 多 Codex 阶段编排

Pipeline Controller 模式借鉴 skill group 思路：父级 controller 负责调度阶段 Codex，每个阶段 Codex 只完成本阶段任务并输出评分卡与交接包。阶段 Codex 不得自行开启下一对话。

### 启用条件

仅当用户明确要求多 Codex、自动开启新对话、pipeline mode、分阶段执行、每个流程单独 Codex 执行等表达时启用。未明确要求 Pipeline 时，仍在当前对话执行同一套结果状态驱动闭环；不得降级为非迭代普通模式。

### Controller 职责

- 维护 `templates/pipeline_state.md`。
- 为每个阶段准备输入包、阶段 prompt、上游产物和禁止事项。
- 若当前环境提供 Codex thread 工具，controller 可以在用户已明确要求 pipeline 时创建下一阶段新对话；若工具不可用，则输出下一阶段 prompt 供用户或当前对话继续执行。
- 读取每个阶段的 `templates/phase_handoff.md` 和 `templates/phase_scorecard.md`。
- 根据阶段评分、质量门、routing tickets 和结果状态决定进入下一阶段、返回修复、停止等待用户事实或最终打包。

### 阶段 Codex 职责

- 只处理 controller 分配的当前阶段任务。
- 必须输出阶段产物、`templates/phase_scorecard.md` 和 `templates/phase_handoff.md`。
- 必须说明阶段总分、结果状态、是否允许进入下一阶段、阻断问题和下一阶段建议。
- 不得自行开启下一 Codex 对话，不得越级完成后续阶段，不得为了推进流程编造事实。

### 标准阶段

标准阶段顺序为：

Intake -> Invention Mining -> Feature Matrix -> Claim Architecture -> Specification / Drawing Plan / Initial Draft -> Review / Routing -> Revision / Regression Decision -> Final Packaging

Review / Routing / Revision / Regression Decision 可以循环，但仍遵守结果状态驱动规则：只有 Regression Decision 输出 REVISE 时才继续修复轮次。

### 阶段评分与进入下一阶段

每个阶段使用 `templates/phase_scorecard.md` 评分。评分不是唯一控制器，必须同时看质量门、事实缺口、编造风险和 routing tickets。

- 阶段 PASS：阶段必需产物完整、无阻断质量门失败、无编造风险，且产物可供下一阶段使用。
- 阶段 REVISE：存在可基于当前材料修复的 Open ticket，且下一步 route 明确。
- 阶段 NEEDS_USER_FACTS：继续推进需要真实事实，AI 不能自主补写。
- 阶段 RISK_ACCEPTED：存在已披露非阻断风险，但可继续进入下一阶段或最终打包。
- 阶段 STOP_LIMIT：达到安全上限或继续调度没有实质收益。

### 多对话交接规则

- 每个阶段完成后，必须先写 handoff，再由 controller 判断是否开启下一对话。
- 新对话必须携带：Case ID、当前 pipeline state、上游 handoff、必须读取的产物、当前 tickets、禁止编造事项和阶段 prompt。
- 新对话完成后，controller 读取其 handoff 和 scorecard，再更新 pipeline state。
- 若达到 STOP_LIMIT 但仍存在 Open Major，不得包装为“最终草案”；只能输出当前工作稿、未通过质量门说明、未关闭 Major ticket 和后续建议。若该 Major 实际依赖用户事实，应转为 Needs User Facts；若为已披露非阻断风险，才可转为 Risk Accepted。

## 6. 评分体系

每轮 review 使用 10 分制，并给出总分、各维度分数、主要扣分原因和需要路由的弱点。

| 维度 | 权重 | 检查重点 |
|---|---:|---|
| Patentability / 可专利性 | 25% | 专利类型、三性、不授权客体、AI/软件技术性、实用新型结构适格性、现有技术压力 |
| Claim Scope / 保护范围合理性 | 20% | 独权宽窄、必要技术特征、从权 fallback、主题布局和绕开风险 |
| Support & Enablement / 说明书支持与充分公开 | 20% | 权利要求支持、实施例、替代方案、参数范围、技术效果因果链 |
| Clarity & Consistency / 清楚性与一致性 | 15% | 术语、编号、引用、段落、主题类型、内部材料一致性 |
| Design-around Resistance / 抗绕开能力 | 10% | 替换部件、调整步骤、改变部署方式、拆分模块等绕开路径 |
| Formality / 中国专利格式合规 | 5% | 权利要求编号、说明书段落、摘要 300 字、附图标记、格式禁忌 |
| Business Alignment / 与用户保护目标匹配 | 5% | 是否覆盖核心产品、替代实现、未来路线和竞品实现方式 |

## 7. 多角色专利审查模拟

Phase 7 必须使用“多角色专利审查模拟”，至少包含：

- 模拟审查员：关注新颖性、创造性、实用性、清楚性、支持性、单一性、不授权客体风险。
- 无效挑战者：关注权利要求是否过宽、说明书是否支持、实施例是否不足、是否容易被现有技术组合击穿。
- 竞争对手设计绕开者：关注是否容易通过替换部件、调整步骤、改变部署方式、拆分模块等方式绕开。
- 专利代理师：关注权利要求层级、从属权利要求 fallback、说明书支撑、术语一致性、撰写规范。
- 形式审查视角：关注名称、段落编号、摘要、附图标记、权利要求编号、从属引用、`[待补充]` 标记。
- 产品/商业保护视角：关注是否覆盖核心产品、替代实现、未来路线和竞品实现方式。

每个角色至少输出：核心发现、严重程度、对应评分扣分、建议路由。不得把模拟审查结论写成真实审查意见、真实检索结论或授权承诺。

## 8. Weakness Routing Table / 弱点路由表

每个审查发现的问题都必须归类，并路由到对应模块重写。

每个 routing ticket 的状态只能是以下之一：Open、Fixed、Deferred、Needs User Facts、Risk Accepted。

- Open 表示尚未处理或修复失败。
- Fixed 表示已修复，并通过回归检查。
- Deferred 表示暂缓处理，但不阻断当前草案输出。
- Needs User Facts 表示需要用户补充真实事实，AI 不能继续自主修复。
- Risk Accepted 表示风险已披露，当前版本接受该风险。

最终输出前必须满足：

- 不允许存在 Open 状态的 Major ticket；
- Major ticket 只能是 Fixed、Needs User Facts 或 Risk Accepted；
- Needs User Facts 必须进入“剩余风险与需用户确认事项”；
- Risk Accepted 必须说明风险内容和接受原因；
- Medium ticket 可以是 Fixed、Deferred、Needs User Facts 或 Risk Accepted；
- Minor ticket 可以保留为 Deferred，但必须披露。

| Weakness | Route | Action |
|---|---|---|
| 独立权利要求过窄 | Claim Architecture | 移除非必要细节，将细节下放到从属权利要求，重建保护范围梯度。 |
| 独立权利要求过宽且说明书支持不足 | Claim Architecture + Specification Support | 收窄权利要求，或增加可由用户材料支持的实施例、替代方案、参数范围；不允许编造事实。 |
| 权利要求包含未在说明书支撑的特征 | Specification Support | 补充说明书支撑段落，或从权利要求删除/降级该特征。 |
| 区别特征和技术效果因果链不足 | Invention Mining + Specification | 重写“技术问题-技术手段-技术效果”闭环，明确哪一特征产生哪一效果。 |
| 现有技术风险高 | Prior Art Pressure Test + Claim Architecture | 做现有技术压力测试，识别可保留的区别特征，调整独权和从权 fallback。 |
| 从属权利要求没有实质 fallback | Claim Architecture | 重新设计从属权利要求层级，使每个从权都提供可用于审查意见答复的收窄位置。 |
| 术语不一致 | Term Table + Specification + Claims | 更新术语统一表，并同步修改权利要求、说明书、摘要和附图建议。 |
| 说明书实施例不足 | Specification + Embodiment Expansion | 基于用户已提供事实补充实施例、变形例、替代方案；若需要用户确认，则标记为 `[待补充]` 或“建议确认”。 |
| AI/算法方案技术性不足 | AI/Software Patentability | 强化具体技术场景、输入数据技术含义、处理流程、模型/模块交互、输出技术结果和技术效果。 |
| 实用新型结构支撑不足 | Utility Model Structure Check + Drawing Plan | 明确产品形状、构造、连接关系和附图绘制要求；如无法支撑实用新型，建议转为发明或补充结构图。 |
| 附图缺失或附图标记不完整 | Drawing Recommendation | 不虚构正式附图，也不把缺图作为默认阻断项；输出建议附图清单、图示内容、图中应包含的部件/模块/步骤/箭头、建议标记、每张图对应的权利要求特征。 |
| 摘要不合规 | Abstract Rewrite | 重写为一个连续段落，不超过 300 字，包含名称、技术领域、技术问题、技术方案要点和用途。 |
| 格式问题 | Formality Check | 修正编号、标题、段落编号、权利要求引用、摘要格式、`[待补充]` 标记等。 |

## 9. Regression Check / 回归检查

每次修复后必须执行回归检查：

- 修复的问题是否关闭；
- 修复是否引入新问题；
- 权利要求修改后说明书是否仍然支持；
- 说明书补充后摘要和术语是否同步；
- 附图建议是否覆盖关键技术特征；
- 从属权利要求引用是否仍然正确；
- 段落编号是否仍然连续；
- 禁止编造规则是否被违反。

如果回归检查发现新增问题，应先生成或更新 routing ticket，再进入 Iteration Decision；只有结果状态为 REVISE 时才继续下一轮。若新增问题仅为非阻断缺失，应保留草案并在最终“剩余风险与需用户确认事项”中列出。

## 10. AI 自主迭代规则

- 尽量在一次响应中完成多轮内部迭代，不应每发现一个问题就要求用户确认。
- 对非阻断缺失，继续生成，并在最终剩余风险中提示。
- 对可以从用户材料合理概括的内容，可以进行专利写作层面的上位概括。
- 对用户未提供的具体技术事实、实验数据、参数、产品结构、正式附图、现有技术文献、检索结论，不能编造。
- 申请人、发明人、代理机构、优先权等主体/程序信息默认不进入技术草案；用户明确要求正式申请信息时，只能照录用户提供内容，不能推断补写。
- 对合理但未确认的替代方案，标记为“建议确认的可选实施方式”，不能写成确定事实。
- 只有当用户输入完全无法识别任何技术方案，或方案明显违法且无法限定到合法技术场景时，才停止生成完整草案并改为输出补充问题或风险提示。

## 11. 专利类型与不授权客体规则

先排除禁止性情形和不授权客体，再判断发明、实用新型或外观设计边界。

- 违反法律、社会公德或妨害公共利益的方案，不按普通可授权方案生成。
- 涉及遗传资源时，检查直接来源、原始来源和合法获取/利用情况；缺失时列为待确认。
- 科学发现、智力活动规则和方法、疾病诊断和治疗方法、动物和植物品种、原子核变换方法及相关物质、主要起标识作用的平面印刷品设计，原则上不按发明/实用新型生成；动物和植物品种的生产方法可继续按发明判断。
- 创新点主要是产品整体或局部形状、图案、色彩及其组合，且目的主要是美感或视觉识别时，提示更接近外观设计。
- 实用新型必须落在产品的形状、构造或者二者结合上；方法、流程、算法、软件、用途、控制策略、材料配方、分子结构、金相结构、自然形成形状、装饰图案，不作为实用新型核心保护对象。
- 发明可覆盖产品、方法或者其改进形成的新的技术方案；软件、算法、AI、大数据方案只有体现为解决技术问题、采用技术手段并获得技术效果时，才按发明组织。
- 结构型产品可提示发明、实用新型或同日双报；同日双报时必须提示同样发明创造只能授予一项专利权，两件申请均需分别说明已申请另一专利，后续发明授权可能需要放弃已授权且未终止的实用新型。

## 12. 现有技术处理规则

本 skill 不默认具备真实检索能力。

- 不得编造现有技术文献、专利号、论文、公开日期或检索结论。
- 用户未提供现有技术时，可以做“现有技术压力测试”：假设本领域常规方案可能包含哪些通用特征，并据此检查权利要求脆弱点；必须标注为“压力测试假设”，不能写成真实检索结果。
- 用户提供现有技术时，必须做 element-by-element 对比，把每个权利要求特征映射到已知现有技术、区别特征或待确认项。
- 现有技术不足时，不停止生成，但在剩余风险中标记为高风险，并建议用户补充检索结果、最接近现有技术和公开情况。

## 13. 权利要求闭环优化规则

- 先生成 Claim Scope Ladder，再写正式权利要求。
- 独立权利要求应尽量只包含解决技术问题的必要技术特征，不遗漏核心技术贡献，也不塞入非必要实施细节。
- 从属权利要求要形成 fallback ladder；每个从属权利要求必须有明确作用：增强创造性、支撑审查意见答复、覆盖优选实施例、防止竞品绕开、固化关键技术效果、支撑实用新型结构限定。
- 不要把所有细节塞进独权；不要只写一个宽泛但无说明书支持的独权。
- 产品权利要求优先写结构、组成、连接关系、模块交互或可确定参数；方法权利要求优先写步骤、流程、操作条件和参数。
- 不在同一项权利要求主题中混合产品和方法。
- 从属权利要求只能引用在前权利要求；多项从属权利要求只能择一引用，且不得作为另一项多项从属权利要求的引用基础。
- 权利要求编号使用 `1.`、`2.`、`3.` 连续编号，编号前不写“权利要求”或“权项”；权利要求书内不加标题，不插图，通常不使用表格。
- 功能性或效果性限定仅在结构/步骤限定不可行或不如功能/效果限定恰当，且说明书能够直接、肯定验证时谨慎使用。
- 附图标记如写入权利要求，应放在对应技术特征后的括号中，并与附图建议/真实附图一致；附图标记仅便于理解，不限制保护范围。

## 14. 说明书闭环优化规则

- 说明书不是简单扩写权利要求；必须为当前权利要求和未来可能修改的 fallback 提供支撑。
- 说明书应包含专利名称、技术领域、背景技术、发明内容/实用新型内容、具体实施方式；有真实附图时加入附图说明，无真实附图时不得把建议附图写成已存在附图。
- 说明书正文段落编号使用 `[0001]`、`[0002]`、`[0003]`……并连续。
- 每个重要权利要求特征都应在具体实施方式中有支撑。
- 上位概括、替代方案、变形例、参数范围、算法步骤、模块交互应尽可能写入说明书，以保留修改空间。
- 技术方案不得只写功能或效果，必须写明实现功能或效果的结构、步骤、参数、模块关系、数据流或交互机制。
- 技术效果应由技术特征直接带来或合理推出；依赖实验结果才能成立的效果必须有用户提供的实验条件、方法、对照对象和数据，否则标记 `[待补充]`。
- 对建议性扩展内容，标注“建议用户确认”，不能写成确定事实。
- 说明书不得使用商业宣传语、无依据贬低语，也不得用“如权利要求……所述”替代具体说明。

## 15. 摘要、术语和禁止编造规则

- 摘要写成一个无标题连续段落，包含名称、技术领域、技术问题、技术方案要点和主要用途；包括标点符号在内不超过 300 字；不得加入说明书和权利要求没有支持的新内容。
- 术语统一表必须同步权利要求、说明书、摘要和附图建议；同一技术对象只能使用一个统一术语。
- 主体/程序信息默认不进入本技术草案，不因申请人、发明人、代理机构、代理师、优先权、分案、同日双报、审阅结论等缺失而降低说明书草案的结果状态。
- 若用户明确要求生成正式申请信息清单或内部材料一致性检查，只能使用用户提供的真实主体/程序信息；缺失时说明“本技术草案未处理正式申请字段”，不得自行补写。
- 禁止把单位、团队、AI、模型或系统名称等未确认内容写作发明人；禁止虚构身份证号、统一社会信用代码、代理机构名称、代理师、申请号或优先权。

## 16. AI、算法、软件、大数据专项规则

- 先判断是否存在具体技术场景、技术问题、技术数据、技术手段和技术效果。
- 不能仅以“模型”“算法”“平台”“推荐”“画像”“智能”等名词证明技术性。
- 仅涉及抽象算法、数学计算规则、计算机程序本身、游戏规则、纯商业规则或管理规则且不包含技术特征的，提示智力活动规则和方法风险。
- 输入数据应具有具体技术领域中的技术含义；输出应为技术结果，而非单纯商业、金融、营销、管理或主观分类结果。
- 技术效果可体现为精度、效率、可靠性、安全性、存储量、传输量、处理速度、硬件执行效率等客观效果。
- 说明书应写明模型模块、层级、连接关系、训练步骤、关键参数、输入数据、输出数据、数据预处理、模型与具体应用场景的结合关系、部署环境、系统交互和技术效果验证或合理说明。
- 方法权利要求可按采集、预处理、特征提取、训练、推理、融合、控制、通信、输出等技术步骤撰写。
- 装置、设备、系统、计算机可读存储介质、计算机程序产品权利要求应与方法步骤或模块交互对应，并得到说明书支持。
- 涉及个人信息、隐私数据、生物识别、人脸识别、用户画像、精准营销、推荐决策、自动驾驶伦理、医疗预测、公共场所图像采集、公共安全等，应输出合法来源、授权同意、必要性、处理范围、脱敏、访问控制、存储/传输安全、隐私保护、公共利益、算法公平性和社会公德风险提示。

## 17. 附图绘制建议规则

附图绘制建议是一等公民。

- skill 不生成正式附图，不虚构已存在的图，不使用“如图1所示”作为未确认事实。
- skill 必须给出建议附图，覆盖权利要求中的关键结构、步骤、模块、数据流或控制关系。
- 对实用新型，优先输出结构示意图、局部放大图、装配关系图、剖视图、爆炸图、线路结构图等建议。
- 对方法/软件/AI 发明，优先输出流程图、系统架构图、模块框图、数据流图、训练流程图、推理流程图、部署架构图、时序图、控制流程图等建议。
- 每张建议图都要说明：为什么需要这张图；支撑哪些权利要求特征；支撑说明书哪个部分；图中应有哪些部件、模块、步骤或箭头；建议使用哪些附图标记；用户绘图时需要确认哪些事实。
- 正式附图缺失不作为默认质量门失败，不单独导致 NEEDS_USER_FACTS；默认交付应包含足够详细的建议图方案，说明每张图要表达什么、包含什么和如何支撑说明书。
- 在用户尚未实际绘制并确认附图前，说明书正式“附图说明”部分不得写成已经存在的图；可输出“建议图 1 为……”和“待用户绘制确认”。
