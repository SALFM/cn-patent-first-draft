# Phase 9：定向修复与重写

## 目标

根据 routing tickets 对草案进行定向重写，关闭审查发现的弱点。

## 输入

- routing tickets。
- 当前版本草案。
- 支持映射、术语表和附图建议。

## 输出

- 修订后的权利要求、说明书、摘要、术语表、附图建议或风险声明。
- 每个 ticket 的处理结果。

## 禁止事项

- 不得为修复问题而编造事实。
- 不得修复一个问题却不更新关联章节。
- 不得把建议附图改写成正式附图。
- 不得修复 `Needs User Facts` ticket，除非用户已经补充对应真实事实。
- 不得为了关闭 ticket 把未确认内容写成确定事实。
- 不得为了补齐格式而擅自加入申请人、发明人、代理机构、优先权等主体/程序信息；这些内容默认不属于技术草案修复对象。

## 质量门

- 每项修改均能对应 ticket。
- 只允许修复状态为 `Open` 且可基于现有材料修复的 ticket。
- 对 `Needs User Facts`、`Risk Accepted`、`Deferred`、`Fixed` ticket 只可保留状态、补充风险说明或更新引用，不得强行改写为已修复。
- 权利要求修改后说明书同步支持。
- 说明书修改后摘要、术语和附图建议同步。
- 附图相关修复应优先补全建议图的目的、图中应包含内容、建议标记和支持关系，而不是要求用户先提供正式附图。
- 每项修改后都要保留可追溯说明：处理了哪个 ticket、修改了哪些章节、对应验收标准是否满足。

## 失败时如何处理

如缺少用户事实导致无法修复，标记 ticket 为 `Needs User Facts`，并在剩余风险中列明所需补充事实。如问题暂缓处理但不阻断当前草案输出，可标记为 `Deferred` 并说明原因。
