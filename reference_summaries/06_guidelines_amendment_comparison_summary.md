# 《专利审查指南》修改对照表摘要：影响本 skill 的新增或修改规则

资料来源：`references/《专利审查指南》修改对照表.pdf`

说明：本文件是修改对照表摘要，不是完整《专利审查指南》摘要。以下仅提取会影响本 skill 生成专利初稿、内部材料检查、专利类型建议、风险提示和自检清单的新增或修改规则。

## 1. 发明人相关修改

### 修改点

- 明确“发明人是指对发明创造的实质性特点作出创造性贡献的人”。
- 新增“不得填写虚假的发明人”。
- 审查员一般不审查请求书中发明人是否符合实质性贡献要求，但有证据表明不符合规定的除外。
- 将“发明人应当是个人”进一步明确为“发明人应当是个人（即自然人）”。
- 新增请求书中应填写所有发明人的身份信息，并确保信息真实。
- 请求书中不得填写单位、集体或人工智能名称，例如课题组、人工智能名称等。

### 修改后的规则

发明人必须是自然人，且必须是对发明创造实质性特点作出创造性贡献的人。请求书应填写所有发明人的真实身份信息，不得填写虚假的发明人，不得将单位、集体、课题组、系统、模型或人工智能名称作为发明人。

### 对 skill 的影响

- 本 skill 不应根据技术交底材料自动臆测发明人。
- 初稿或内部材料中如需涉及发明人，应提示由申请人确认真实发明人名单和身份信息。
- 对“AI 参与研发”“课题组完成”“团队完成”等描述，应区分技术来源说明与发明人填写要求：AI、单位、课题组不能作为发明人。
- 如用户提供的发明人字段包含公司、部门、课题组、项目组、算法模型、AI 系统名称，应在风险提示中标红或重点提示。

### 应写入 SKILL.md 的内容

- 在信息收集阶段增加“发明人信息核验”要求：发明人应为自然人，且应为对发明创造实质性特点作出创造性贡献的人。
- 生成请求书相关提示或内部交付材料时，不得将单位、集体、课题组、AI 系统、算法模型名称写作发明人。
- 若发明人信息缺失、疑似虚假、仅提供单位/团队/AI 名称，应输出待确认项和风险提示，不得默认补全。
- 对 AI 辅助完成的技术方案，应提醒确认自然人发明人及其创造性贡献。

### 应写入 self_checklist.md 的内容

- [ ] 发明人是否均为自然人？
- [ ] 是否已确认所有发明人均对发明创造的实质性特点作出创造性贡献？
- [ ] 是否已填写所有发明人的真实身份信息？
- [ ] 发明人字段中是否不存在单位、部门、课题组、项目组、AI、算法模型或系统名称？
- [ ] 是否已提示不得填写虚假的发明人？

### 风险提示

发明人信息不真实、遗漏真实发明人、填写单位/集体/AI 名称，可能导致申请文件存在著录事项风险、权属争议风险、诚信风险或后续程序风险。对于 AI 辅助研发项目，尤其应提示“AI 不能作为发明人，需确认作出创造性贡献的自然人”。

## 2. 代理机构相关修改

### 修改点

- 专利代理机构名称仍应使用其在国家知识产权局登记的全称，并与申请文件中加盖的代理机构公章名称一致，不得使用简称或缩写。
- 新增：专利代理机构应当对请求书中填写的申请人身份信息和联系方式进行核实。
- 新增：专利代理机构或者专利代理师以自己的名义申请专利或请求宣告专利权无效的，依照《专利代理条例》处理。

### 修改后的规则

代理机构名称应使用登记全称，不得使用简称或缩写，并应与公章一致。代理机构还负有核实申请人身份信息和联系方式的义务。

### 对 skill 的影响

- 本 skill 生成内部材料检查项时，应把“代理机构登记全称、机构代码、公章名称一致性”作为形式检查项。
- 对申请人身份信息、统一社会信用代码/身份证明、地址、联系人、电话、邮箱等信息，应提示由代理机构或经办人核实。
- 本 skill 不应将代理机构简称直接写入正式请求书相关字段；若用户只提供简称，应提示补充登记全称。

### 应写入 SKILL.md 的内容

- 在内部材料检查中增加“代理机构信息核验”：名称使用国家知识产权局登记全称，机构代码准确，公章名称一致，不使用简称或缩写。
- 在申请人信息检查中增加“代理机构应核实申请人身份信息和联系方式”的提示。
- 若代理机构名称疑似简称、品牌名、英文缩写或与公章不一致，应输出待确认项。

### 应写入 self_checklist.md 的内容

- [ ] 代理机构名称是否为国家知识产权局登记全称？
- [ ] 代理机构名称是否与申请文件公章一致？
- [ ] 是否已填写正确的代理机构代码？
- [ ] 申请人身份信息和联系方式是否已核实？
- [ ] 是否避免使用代理机构简称、缩写、品牌名替代登记全称？

### 风险提示

代理机构名称不规范、机构代码错误、公章名称不一致，可能导致形式审查补正或程序文件不规范。申请人身份信息或联系方式未经核实，可能影响通知送达、权属确认、费用缴纳和后续程序稳定性。

## 3. 优先权和分案申请相关修改

### 修改点

- 新增：分案申请的原申请要求了优先权，但申请人提出分案申请时未在请求书中声明要求该优先权的，分案申请视为未要求该优先权。
- 审查员应发出视为未要求优先权通知书。

### 修改后的规则

分案申请不能当然继承原申请的优先权声明。原申请要求优先权的，分案申请提出时仍需要在请求书中声明要求该优先权；未声明的，分案申请视为未要求该优先权。

### 对 skill 的影响

- 仅作为风险提醒写入本 skill，不展开为分案申请办案程序。
- 当用户材料显示属于分案申请、延续原申请技术内容、或提及原申请优先权时，应提醒检查分案请求书是否单独声明优先权。
- 不应默认写成“分案自动享有原申请优先权”。

### 应写入 SKILL.md 的内容

- 在风险识别规则中增加：若案件为分案申请且原申请要求优先权，应提示“分案申请提出时需在请求书中声明该优先权，否则分案申请可能被视为未要求该优先权”。
- 该项仅作为风险提示，不在本 skill 中展开优先权恢复、补正、期限计算等办案程序。

### 应写入 self_checklist.md 的内容

- [ ] 本案是否为分案申请？
- [ ] 原申请是否要求优先权？
- [ ] 若原申请要求优先权，分案申请请求书是否已声明要求该优先权？
- [ ] 是否已提示未声明可能导致分案申请被视为未要求优先权？

### 风险提示

分案申请未声明原申请优先权，可能导致分案申请被视为未要求优先权，进而影响新颖性、创造性评价中的现有技术日期判断。该风险应作为初稿交付时的醒目程序风险提醒。

## 4. 同日申请发明和实用新型相关修改

### 修改点

- 对同一申请人同日就同样的发明创造既申请实用新型又申请发明专利的，新增/明确应根据专利法实施细则第四十七条，在申请时分别说明已对同样的发明创造申请了另一专利。
- 未作说明的，依照专利法第九条第一款关于同样的发明创造只能授予一项专利权的规定处理。
- 作出说明且发明申请符合授权条件的，后续可能需要在规定期限内声明放弃已授权的实用新型专利权，以避免重复授权。

### 修改后的规则

同一申请人同日对同样的发明创造同时申请发明和实用新型，应在两件申请提交时分别说明已申请另一专利。若后续发明申请可以授权且在先实用新型仍有效，申请人通常需要选择修改或声明放弃已授权的实用新型，以避免重复授权。

### 对 skill 的影响

- 在“专利类型建议”中，若建议同日提交发明和实用新型，应同步提示分别说明义务。
- 在“风险提示”中，提示后续发明授权阶段可能需要放弃已授权实用新型，用户应权衡保护期限、授权速度、稳定性和商业安排。
- 不能只建议“双报”而不提示同样发明创造只能授予一项专利权及放弃实用新型的后续风险。

### 应写入 SKILL.md 的内容

- 在专利类型建议规则中增加：如建议“同日发明+实用新型”布局，应提示申请时两件申请均需分别说明已对同样的发明创造申请另一专利。
- 增加双报风险说明：发明申请后续符合授权条件时，若实用新型已授权且未终止，可能需要声明放弃实用新型或修改发明申请以避免重复授权。
- 对用户只要求“快速保护”时，可建议实用新型；对核心技术和长期保护价值较高的方案，可提示考虑发明或同日双报，但必须附带重复授权风险。

### 应写入 self_checklist.md 的内容

- [ ] 是否建议同日申请发明和实用新型？
- [ ] 若建议同日双报，是否已提示两件申请在申请时分别说明已申请另一专利？
- [ ] 是否已提示同样的发明创造只能授予一项专利权？
- [ ] 是否已提示后续发明授权时可能需要放弃已授权实用新型或修改发明申请？

### 风险提示

同日双报可以兼顾较快授权和较长期限保护，但若未在申请时分别说明，或后续不愿放弃已授权实用新型，可能影响发明授权路径。该事项应写入专利类型建议和交付风险提示，不宜作为单纯“推荐双报”的无条件建议。

## 5. 创造性审查相关修改

### 修改点

- 明确创造性评价应针对权利要求限定的技术方案整体进行，而不是评价某一技术特征本身是否具备创造性。
- 强调对现有技术作出贡献的技术特征应写入权利要求；否则即使说明书中有记载，评价创造性时也不予考虑。
- 新增：对技术问题的解决没有作出贡献的特征，即使写入权利要求中，通常也不会对技术方案的创造性产生影响。

### 修改后的规则

创造性审查以权利要求限定的技术方案整体为对象。能够解决技术问题、产生技术效果、带来预料不到效果或克服技术偏见的技术特征，应进入权利要求。与技术问题和技术效果无关的附加特征，通常不能正面支撑创造性。

### 对 skill 的影响

- 本 skill 生成权利要求时，应优先写入对技术问题、技术手段、技术效果有贡献的技术特征。
- 不应为了“看起来更具体”而堆砌外壳形状、界面样式、位置关系、常规模块等与技术问题无关的特征。
- 说明书中的发明点、技术效果和权利要求限定内容应保持对应关系，避免关键贡献只写在说明书而未进入权利要求。
- 在整理技术交底时，应提取“区别特征-解决的技术问题-产生的技术效果”链条，并据此组织独立权利要求和从属权利要求。

### 应写入 SKILL.md 的内容

- 权利要求撰写规则应增加：独立权利要求优先限定解决技术问题并产生技术效果的必要技术特征。
- 从属权利要求应围绕进一步技术效果展开，不将与技术问题无关的常规部件、外观、业务标签作为主要创造性支撑。
- 生成创造性论述时，应以权利要求限定的技术方案整体评价，并说明各贡献特征如何协同解决技术问题。
- 若用户提供的“创新点”只是无技术贡献的描述，应提示其通常难以支撑创造性。

### 应写入 self_checklist.md 的内容

- [ ] 独立权利要求是否写入了对技术问题有贡献的必要技术特征？
- [ ] 说明书中的核心技术贡献是否已体现在权利要求中？
- [ ] 每个主要区别特征是否能对应至少一个技术问题或技术效果？
- [ ] 是否删除或弱化了与技术问题无关、仅用于堆砌限定的特征？
- [ ] 创造性分析是否以权利要求限定的技术方案整体为对象？

### 风险提示

仅在说明书中描述技术贡献而未写入权利要求，可能导致创造性评价时不被考虑。向权利要求加入与技术问题无关的特征，通常不能补强创造性，反而可能不必要地限缩保护范围。

## 6. AI、大数据、算法和商业规则相关修改

### 修改点

- 章节标题从“包含算法特征或商业规则和方法特征的发明专利申请”扩展为“涉及人工智能、大数据等包含算法特征或商业规则和方法特征的发明专利申请”。
- 审查时不应简单割裂技术特征与算法特征、商业规则和方法特征，应将权利要求记载的内容作为整体，分析技术手段、解决的技术问题和获得的技术效果；必要时还应结合说明书内容审查。
- 新增根据专利法第五条第一款的审查：包含算法特征或商业规则和方法特征的申请中，如数据采集、标签管理、规则设置、推荐决策等含有违反法律、社会公德或者妨害公共利益的内容，不能被授予专利权。
- 新增/调整示例强调：将图像采集、人脸识别、个人身份识别用于商场精准营销，且未表明数据获取或信息采集合法合规的，存在不能授权风险。
- 新增/调整示例强调：无人驾驶应急决策模型若以行人性别、年龄决定被保护对象和被撞对象，可能违反生命平等伦理、强化偏见、引发公共安全信任问题，属于违反社会公德内容，不能授予专利权。
- 说明书撰写部分新增：涉及人工智能模型构建或训练的，一般需清楚记载模型必要模块、层级或连接关系，以及训练必需的具体步骤、参数等；涉及具体领域或场景中应用 AI 模型或算法的，一般需清楚记载模型/算法如何与具体场景结合，输入、输出数据如何设置并体现内在关联。

### 修改后的规则

AI、大数据、算法和商业规则类方案应整体判断是否通过技术手段解决技术问题并获得技术效果。算法特征、商业规则特征如果与技术特征在功能上彼此支持、存在相互作用，可以纳入技术方案贡献评价。但若数据采集、标签、规则、推荐、决策等内容违反法律、社会公德或妨害公共利益，即使包含算法或技术模块，也不能被授予专利权。

### 对 skill 的影响

- 本 skill 处理 AI、大数据、推荐系统、用户画像、风控、自动驾驶、医疗预测、人脸识别等主题时，应整体构建“技术问题-技术手段-技术效果”链条。
- 不应把算法特征简单排除在技术方案外；应判断算法是否与具体技术场景、数据处理、设备控制、资源利用、识别准确率、系统性能等技术效果存在内在关联。
- 说明书应具体写明模型结构、训练步骤、参数、输入输出数据、数据处理流程、模型与应用场景的结合方式，避免只写抽象算法、商业目标或结果。
- 对涉及个人信息、生物识别、人脸识别、用户画像、精准营销、自动驾驶伦理决策等方案，应自动增加合法性、必要性、数据来源、授权同意、公共利益、隐私保护、公平性和伦理风险提示。

### 应写入 SKILL.md 的内容

- 对 AI/算法/大数据类方案，先判断是否存在具体技术场景、技术问题、技术手段和技术效果，再组织权利要求与说明书。
- 算法特征或商业规则特征不得被简单割裂；若其与技术特征功能上相互支持并产生技术效果，应在权利要求和说明书中体现这种相互作用。
- 说明书生成规则应要求写明：模型模块、层级、连接关系、训练步骤、关键参数、输入数据、输出数据、数据预处理、模型与具体应用场景的结合关系、技术效果验证或合理说明。
- 合规风险规则应增加：涉及个人信息、人脸识别、生物识别、用户画像、精准营销、自动驾驶伦理决策、医疗预测、公共场所图像采集、推荐决策时，必须输出合法性、必要性、数据来源、授权同意、隐私保护、公共利益、算法公平性和社会公德风险提示。
- 对违反法律、社会公德或妨害公共利益的算法/数据方案，应提示存在不能被授予专利权的实质性风险。

### 应写入 self_checklist.md 的内容

- [ ] AI/算法/大数据方案是否明确了具体技术问题？
- [ ] 算法特征是否与技术特征功能上相互支持、存在相互作用？
- [ ] 权利要求是否整体体现技术手段、技术问题和技术效果，而非只描述商业规则或抽象算法？
- [ ] 说明书是否写明模型模块、层级、连接关系、训练步骤、关键参数？
- [ ] 是否写明输入数据、输出数据及其与具体场景的内在关联？
- [ ] 是否说明数据来源、采集方式、授权同意或合法合规基础？
- [ ] 是否涉及个人信息、人脸识别、生物识别、用户画像、精准营销、自动驾驶伦理决策、医疗预测或公共场所图像采集？
- [ ] 若涉及上述敏感场景，是否已增加合法性、必要性、公共利益、隐私保护、公平性和社会公德风险提示？
- [ ] 是否排除了违反法律、社会公德或妨害公共利益的规则设置、标签管理、推荐决策或自动化决策内容？

### 风险提示

AI、算法或大数据方案不能仅依靠“模型”“算法”“平台”“推荐”“画像”等名词获得技术性或创造性支撑。若方案本质是商业规则、管理规则或抽象算法，且未体现技术手段、技术问题和技术效果，可能不属于可授权技术方案。若涉及违法采集个人信息、未经必要性论证的人脸识别、隐蔽式用户画像、精准营销、基于年龄/性别等敏感因素的自动驾驶伦理决策，可能因违反法律、社会公德或妨害公共利益而不能被授予专利权。

## 7. 汇总写入建议

### 修改点

本修改对照表对本 skill 的影响集中在五类输出：发明人和代理机构形式核验、优先权/分案风险提醒、同日发明与实用新型布局提示、创造性导向的权利要求生成、AI/大数据/算法/商业规则方案的技术性和合规风险识别。

### 修改后的规则

本 skill 应把“真实主体信息、程序声明风险、技术贡献进入权利要求、AI/算法方案整体技术性审查、数据与伦理合规”作为生成专利初稿时的强制检查维度。

### 对 skill 的影响

- 信息收集：增加发明人、申请人、代理机构、优先权、分案、双报状态的待确认项。
- 初稿生成：权利要求优先表达对技术问题和技术效果有贡献的技术特征。
- 专利类型建议：建议同日发明+实用新型时必须提示分别说明和后续放弃实用新型风险。
- 风险提示：对分案优先权、AI/个人信息/人脸识别/精准营销/自动驾驶伦理决策给出明确风险。
- 自检清单：把主体信息真实性、权利要求技术贡献、AI 数据合规列为必检项。

### 应写入 SKILL.md 的内容

应把上述规则分别写入信息收集、权利要求撰写、说明书撰写、专利类型建议、内部材料检查、风险提示和交付前自检流程中。写入时应作为操作规则和检查项，不应把本修改对照表改写成完整审查指南摘要。

### 应写入 self_checklist.md 的内容

self_checklist.md 应新增或合并以下检查组：

- 发明人真实性与自然人身份检查。
- 代理机构登记全称、机构代码、公章一致性、申请人身份和联系方式核实。
- 分案申请优先权声明风险检查。
- 同日发明和实用新型双报说明及后续放弃实用新型风险检查。
- 创造性贡献特征是否进入权利要求检查。
- AI/算法/大数据/商业规则方案的技术性、充分公开、数据来源、个人信息保护、公共利益和伦理风险检查。

### 风险提示

本修改对照表反映出审查规则对真实性、合规性和技术贡献对应关系的要求更细。后续使用本 skill 生成初稿时，应避免只输出撰写文本而不提示主体信息、程序声明、重复授权、数据合规和伦理风险；尤其不能把 AI、单位、课题组写成发明人，不能把分案优先权和同日双报风险默认视为无风险事项，不能用与技术问题无关的特征堆砌创造性。
